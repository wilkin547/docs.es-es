## <a name="service"></a><span data-ttu-id="df0cc-116">Servicio</span><span class="sxs-lookup"><span data-stu-id="df0cc-116">Service</span></span>  

 <span data-ttu-id="df0cc-117">El servicio en este ejemplo tiene dos contratos, `ICalculator` e `IEcho`.</span><span class="sxs-lookup"><span data-stu-id="df0cc-117">The service in this sample has two contracts, `ICalculator` and `IEcho`.</span></span> <span data-ttu-id="df0cc-118">Además del punto de conexión `IMetadataExchange` de costumbre, hay tres puntos de conexión de la aplicación, tal y como se muestra en el código siguiente.</span><span class="sxs-lookup"><span data-stu-id="df0cc-118">In addition to the customary `IMetadataExchange` endpoint, there are three application endpoints, as shown in the following code.</span></span>

```xml  
        binding="wsHttpBinding"  
        contract="Microsoft.ServiceModel.Samples.ICalculator"

        binding="wsHttpBinding"
```  

 <span data-ttu-id="df0cc-119">Los tres extremos se hospedan en el mismo `ListenUri` y utilizan el mismo `binding`, los extremos en el mismo `ListenUri` deben tener el mismo enlace, porque están compartiendo una pila de canal única que realiza escuchas para los mensajes en esa dirección física en el equipo.</span><span class="sxs-lookup"><span data-stu-id="df0cc-119">All three endpoints are hosted at the same `ListenUri` and use the same `binding` - endpoints at the same `ListenUri` must have the same binding, because they are sharing a single channel stack that listens for messages at that physical address on the machine.</span></span> <span data-ttu-id="df0cc-120">La`address` de cada punto de conexión es un URN; aunque normalmente las direcciones representan las ubicaciones físicas, de hecho la dirección puede ser cualquier tipo de URI, porque la dirección se utiliza para coincidir y filtrar los propósitos como se muestra en este ejemplo.</span><span class="sxs-lookup"><span data-stu-id="df0cc-120">The `address` of each endpoint is a URN; though typically addresses represent physical locations, in fact the address can be any kind of URI, because the address is used for matching and filtering purposes as is demonstrated in this sample.</span></span>  
  
 <span data-ttu-id="df0cc-121">Dado que los tres puntos de conexión comparten el mismo `ListenUri` , cuando un mensaje llega allí, Windows Communication Foundation (WCF) debe decidir a qué punto de conexión se destina el mensaje.</span><span class="sxs-lookup"><span data-stu-id="df0cc-121">Because all three endpoints share the same `ListenUri`, when a message arrives there, Windows Communication ## <a name="client"></a><span data-ttu-id="df0cc-131">Cliente</span><span class="sxs-lookup"><span data-stu-id="df0cc-131">Client</span></span>  

 <span data-ttu-id="df0cc-132">Así como los puntos de conexión en el servidor tienen dos direcciones diferentes, los puntos de conexión del cliente también tienen dos direcciones.</span><span class="sxs-lookup"><span data-stu-id="df0cc-132">Just as endpoints on the server have two different addresses, client endpoints also have two addresses.</span></span> <span data-ttu-id="df0cc-133">En el servidor y el cliente, la dirección lógica se denomina `EndpointAddress`.</span><span class="sxs-lookup"><span data-stu-id="df0cc-133">On both server and client, the logical address is called the `EndpointAddress`.</span></span> <span data-ttu-id="df0cc-134">Pero mientras que la dirección física se denomina `ListenUri` en el servidor, en el cliente, la dirección física se denomina `Via`.</span><span class="sxs-lookup"><span data-stu-id="df0cc-134">But whereas the physical address is called the `ListenUri` on the server, on the client, the physical address is called the `Via`.</span></span>  
  
 <span data-ttu-id="df0cc-135">Como en el servidor, de forma predeterminada, estas dos direcciones son las mismas.</span><span class="sxs-lookup"><span data-stu-id="df0cc-135">As on the server, by default, these two addresses are the same.</span></span> <span data-ttu-id="df0cc-136">Para especificar una`Via` en el cliente que sea diferente de la dirección del extremo, `ClientViaBehavior` se utiliza:</span><span class="sxs-lookup"><span data-stu-id="df0cc-136">To specify a `Via` on the client that is different from the endpoint's address, `ClientViaBehavior` is used:</span></span>

```csharp  
calcClient.ChannelFactory.Endpoint.Behaviors.Add(  
        new ClientViaBehavior(via));  
```  
  
 <span data-ttu-id="df0cc-137">Como de costumbre, la dirección procede del archivo de configuración del cliente, que fue generado por Svcutil.exe.</span><span class="sxs-lookup"><span data-stu-id="df0cc-137">As usual, the address comes from the client configuration file, which was generated by Svcutil.exe.</span></span> <span data-ttu-id="df0cc-138">`Via` (Qué corresponde a `ListenUri` del servicio) no aparece en los metadatos del servicio y así esta información se debe comunicar al cliente fuera de banda (simplemente como la dirección de los metadatos del servicio).</span><span class="sxs-lookup"><span data-stu-id="df0cc-138">The `Via` (which corresponds to the `ListenUri` of the service) does not appear in the service's metadata and so this information must be communicated to the client out-of-band (just like the service's metadata address).</span></span>  
  
 <span data-ttu-id="df0cc-139">El cliente en este ejemplo envía los mensajes a cada uno de los tres puntos de conexión de la aplicación del servidor, para mostrar que puede comunicarse con (y diferenciar) los tres puntos de conexión, aunque ellos todos tienen el mismo `Via`.</span><span class="sxs-lookup"><span data-stu-id="df0cc-139">The client in this sample sends messages to each of the server's three application endpoints, to demonstrate that it can communicate with (and differentiate) all three endpoints, even though they all have the same `Via`.</span></span>
